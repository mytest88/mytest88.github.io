Active Directory Threat Simulation and SIEM Analysis

In this project, I built and operated a comprehensive virtual lab environment to simulate Active Directory (AD) threats and analyze their impact using a Security Information and Event Management (SIEM) system. [cite: 1] The primary challenge was to understand how advanced attacks target organizations and to confirm that our security monitoring (SIEM) could effectively capture and detect these threats. [cite: 2] I set up a realistic domain with a Windows Server acting as an Active Directory Domain Controller, a Windows 10 endpoint, and a Kali Linux machine for offensive operations. [cite: 3] All network and system events were systematically collected and analyzed by Splunk, which I deployed on an Ubuntu server. [cite: 4]

Through this lab, I demonstrated my ability to not only simulate common and advanced attack vectors, such as RDP brute-force and Atomic Red Team techniques (like local account creation and data exfiltration), but also to precisely track and analyze their telemetry in Splunk. [cite: 5] This project provided crucial insights into how a well-configured SIEM provides exceptional visibility, even when attacks are unsuccessful. [cite: 6] I confirmed that Splunk effectively captured and indexed critical security events, providing crucial visibility into unauthorized access attempts and simulated adversary activities, which is fundamental for effective threat detection and incident response. [cite: 7]

Key Skills Demonstrated:

Active Directory (AD DS), Splunk SIEM, Universal Forwarders, Sysmon, Kali Linux, Hydra, Atomic Red Team, PowerShell, network configuration, log analysis, threat simulation, incident detection, troubleshooting, forensic evidence collection, threat detection and incident response. [cite: 8]

Workflow

This comprehensive workflow allowed me to simulate realistic attack scenarios and, crucially, to validate and demonstrate the effectiveness of Splunk as a SIEM solution in detecting, logging, and providing deep visibility into Active Directory-focused threats. [cite: 9] Here's a breakdown of the workflow I followed: [cite: 10]

Virtual Lab Environment Setup

I deployed essential virtual machines: a Windows Server as our Active Directory Domain Controller (192.168.10.15), a Windows 10 Endpoint (192.168.10.10) for user activity and attacks, and a Kali Linux machine (192.168.10.13) for offensive tools. [cite: 10] I configured an Ubuntu Server (192.168.10.12) to host our Splunk SIEM. [cite: 11]

Splunk SIEM Deployment and Configuration

I installed Splunk Enterprise on the Ubuntu server, ensuring it was configured with a static IP and set to start automatically on boot. [cite: 12] I deployed Splunk Universal Forwarders on both the Windows Server (AD DC) and the Windows 10 Endpoint. [cite: 13] I configured the input.conf files on the Windows machines to forward critical Windows Event Logs (Application, Security, System, and Sysmon Operational logs) to Splunk. [cite: 14] In Splunk, I created dedicated indexes, endpoint and ADserver, to organize the incoming logs and configured the receiving port (9997) to accept data from the forwarders. [cite: 15] I verified successful log ingestion by searching for events from both Windows machines in Splunk. [cite: 16]

Active Directory Domain Setup

I installed Active Directory Domain Services (AD DS) on the Windows Server and promoted it to a domain controller, creating the hanlab.local forest. [cite: 17] I created new domain user accounts within Active Directory. [cite: 18] I configured the Windows 10 endpoint's DNS settings to point to the AD server and successfully joined it to the hanlab.local domain. [cite: 18]

Threat Simulation: RDP Brute-Force Attack

I prepared a password list on the Kali machine, including the target user's password. [cite: 19] I enabled Remote Desktop on the Windows 10 endpoint and granted RDP access to our domain users. [cite: 20] I executed an RDP brute-force attack using Hydra from the Kali machine against the Windows 10 endpoint. [cite: 21] I troubleshooted an initial connection error by disabling Network Level Authentication (NLA) on the target. [cite: 22] I analyzed the failed logon attempts (Event Code 4625) in Splunk, successfully identifying the Kali machine's IP as the source of the attack, demonstrating Splunk's ability to track malicious activity. [cite: 23]

Threat Simulation: Atomic Red Team Attacks

I prepared the Windows 10 endpoint for Atomic Red Team by adjusting the PowerShell execution policy and configuring Windows Defender exclusions. [cite: 24] I installed Atomic Red Team on the Windows 10 endpoint. [cite: 25] I executed T1136.001 (Create Account - Local Account) and verified the creation of new local user accounts by analyzing corresponding events in Splunk. [cite: 26] I executed T1041 (C2 Data Exfiltration) to simulate data exfiltration attempts. [cite: 27] Even though the exfiltration was blocked, I analyzed the generated telemetry in Splunk, highlighting the critical importance of seeing command-line arguments and attempted data in logs for early warning and forensic analysis. [cite: 28]

Configuring Windows Event Log Inputs for Splunk

To ensure comprehensive log collection from the Windows machines, I modified the `input.conf` file located at `C:\Program Files\SplunkUniversalForwarder\etc\system\local`. [cite: 57] This configuration directs the Splunk Universal Forwarder to collect specific Windows Event Logs and forward them to our Splunk indexer. [cite: 58] As seen in the configuration, I enabled the collection of: [cite: 59]

  • Application Logs (`WinEventLog://Application`) [cite: 59]
  • Security Logs (`WinEventLog://Security`) [cite: 59]
  • System Logs (`WinEventLog://System`) [cite: 59]
  • Sysmon Operational Logs (`WinEventLog://Microsoft-Windows-Sysmon/Operational`) [cite: 59]

For each of these, I set `index = endpoint` to route the data to a dedicated index in Splunk, and `disabled = false` to ensure they are actively collected. [cite: 59] Notably, for Sysmon logs, I also set `renderXml = true` and `source = XmlWinEventLog:Microsoft-Windows-Sysmon/Operational` to ensure the detailed XML structure of Sysmon events is preserved for richer analysis in Splunk. [cite: 60] This setup is crucial for gaining deep visibility into endpoint activities. [cite: 61]

Creating a New Index in Splunk for Endpoint Logs

To properly store and analyze the logs collected from the Windows Universal Forwarders, I created a new dedicated index in Splunk. [cite: 62] I navigated to "Settings" -> "Indexes" and clicked "New Index." [cite: 63] As shown, I named this new index `endpoint`. [cite: 63] This ensures that all the Windows event logs (Application, Security, System, and Sysmon) are stored in a specific, organized location within Splunk, making it easier to search, analyze, and manage the data. Troubleshooting RDP Brute-Force: Network Level Authentication (NLA)

When I first attempted the RDP brute-force attack using Hydra, I encountered an error: "account on 192.168.10.10 might be valid but account not active for remote desktop". [cite: 90] This indicated a successful username enumeration but a failure to establish the RDP connection. [cite: 91] The problem was due to Network Level Authentication (NLA) being enabled on the Windows 10 target. [cite: 92] To resolve this, I navigated back to the "Remote" tab in System Properties on the Windows 10 machine. [cite: 93] As shown, I unchecked the option "Only allow connections from computers running Remote Desktop with Network Level Authentication (recommended)". [cite: 94] Disabling NLA allowed Hydra to proceed with the brute-force attempts without this initial authentication hurdle, enabling the simulation to continue as planned. [cite: 95]

Successful RDP Brute-Force with Hydra

After troubleshooting the Network Level Authentication (NLA) issue, I re-ran the Hydra brute-force attack against the Windows 10 endpoint's RDP service. [cite: 96] As seen in the output, Hydra successfully identified the correct password for the `micardi` user on 192.168.10.10. [cite: 97]

Troubleshooting Atomic Red Team: Command Not Recognized

When I attempted to execute the Atomic Red Team technique T1136.001 (Create Account - Local Account) using `Invoke-AtomicTest T1136.001`, I encountered an error stating, "The term 'Invoke-AtomicTest' is not recognized as the name of a cmdlet, function, script file, or operable program". [cite: 113] This indicates that PowerShell cannot find the `Invoke-AtomicTest` function, likely because the Atomic Red Team module is not properly loaded or its path is not in the PowerShell session's `$env:PSModulePath`. [cite: 114] To resolve this, I need to ensure that the PowerShell session is aware of where the `Invoke-AtomicTest` function resides. [cite: 115] This typically involves importing the module or navigating to the directory where it's located before executing the command. [cite: 116] I will provide the correct path to `Invoke-AtomicTest` to ensure the command runs successfully. [cite: 117]

Executing Atomic Red Team: T1136.001 (Create Account - Local Account)

To resolve the "command not recognized" error and successfully execute Atomic Red Team techniques, I explicitly imported the `Invoke-AtomicRedTeam` module. [cite: 118] I ran `Import-Module "C:\AtomicRedTeam\invoke-atomicredteam\Invoke-AtomicRedTeam.psd1"` to load the module into the PowerShell session. [cite: 119] Following this, I re-executed `Invoke-AtomicTest T1136.001` to simulate creating a local account. [cite: 120] There are some various sub-tests within this technique. [cite: 120] While one test initially failed due to password policy requirements, other tests, such as "Create a new user in PowerShell" and "Create a new Windows admin user," completed successfully, as indicated by `Exit code: 0` and messages like "The command completed successfully". [cite: 121] This demonstrates that Atomic Red Team is now correctly installed and capable of performing simulated attacks within our lab environment. [cite: 122]

Analyzing Data Exfiltration Attempts in Splunk

After simulating the T1041 (Data Exfiltration over C2) Atomic Red Team technique, I immediately turned to Splunk to analyze the resulting telemetry. [cite: 137] I performed a search using `index="endpoint" example.com` to specifically look for any activity related to the attempted exfiltration to `example.com`. [cite: 138]

Here we can see that Splunk successfully captured 4 events related to this activity, even though the actual exfiltration attempt previously resulted in an "Access Denied" error. [cite: 139] This is incredibly important. The logs, specifically Sysmon events, provide crucial details: [cite: 139]

  • They show the `CommandLine` used, indicating `powershell.exe` attempting to connect to `example.com`. [cite: 140]
  • They contain fields like `exfiltratedData` and `encodedData`, revealing the content that was attempted to be exfiltrated (e.g., "SecretDataToExfiltrate" in its encoded form). [cite: 141]

The ability to see this telemetry in Splunk is vital for the analyst. [cite: 142] Even if an attack is unsuccessful due to network controls or misconfigurations, the fact that our SIEM (Splunk) and endpoint logging (Sysmon via Universal Forwarder) capture these attempts provides: [cite: 143]

  • **Early Warning:** These logs serve as critical indicators of compromise (IOCs) and an attacker's intent, allowing us to detect malicious activity before it succeeds. [cite: 143]
  • **Forensic Evidence:** The detailed information within these logs (source IP, command line, attempted data) is invaluable for post-incident analysis and understanding the adversary's techniques. [cite: 144]
  • **Validation of Controls:** It confirms that our logging infrastructure is functioning correctly and that our SIEM is effectively collecting and indexing the necessary data to identify sophisticated attacks like data exfiltration attempts. [cite: 145]
  • **Opportunity for Refinement:** This telemetry can be used to develop or refine detection rules, ensuring that future, similar attempts are immediately flagged and alerted upon. [cite: 146]

This demonstrates the power of a well-configured SIEM in providing deep visibility into endpoint activities, regardless of the attack's success, which is fundamental for effective threat detection and incident response. [cite: 147]
